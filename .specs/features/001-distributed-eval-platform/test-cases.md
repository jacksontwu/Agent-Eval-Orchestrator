## 需求 FR-001: 统一任务模型

**需求描述**: 平台必须提供与执行器解耦的 `TaskTemplate/Run/Batch/CaseRun` 模型。

### 正向场景（必选）

- [ ] **TC-FR001-001**: 创建一个 Harbor 类型任务模板并生成 Run/Batch
  - **前置条件**: controller 已启动
  - **测试步骤**:
    1. 创建任务模板，指定 `executorKind=harbor-docker`
    2. 创建 Run
    3. 发起 Batch
  - **预期结果**: 核心数据结构不暴露 Harbor 专有状态字段作为主状态
  - **验证状态**: ⏳ 待验证
  - **备注**: 核对数据库 schema 与 API 响应

- [ ] **TC-FR001-002**: 使用相同模型创建非 Harbor 执行器任务模板
  - **前置条件**: 存在第二个伪执行器或 mock executor
  - **测试步骤**:
    1. 创建 `executorKind=mock` 的任务模板
    2. 发起 Batch
  - **预期结果**: 平台核心流程无需修改即可接受不同执行器
  - **验证状态**: ⏳ 待验证
  - **备注**: 验证平台内核解耦

### 边界场景（可选）

- [ ] **TC-FR001-EDGE-001**: executorMetadata 缺失可选字段
  - **前置条件**: 已定义最小执行器配置
  - **测试步骤**:
    1. 仅提交最小执行器配置
    2. 发起模板保存
  - **预期结果**: 核心模型仍可创建，缺失字段由执行器层校验
  - **验证状态**: ⏳ 待验证 / ⏭️ 跳过
  - **备注**: 防止平台层错误兜底

## 需求 FR-002: 固定 worker 调度

**需求描述**: worker 必须支持注册、心跳、slots、领取任务和后台执行。

### 正向场景（必选）

- [ ] **TC-FR002-001**: 两台固定 worker 注册后按 slots 领取任务
  - **前置条件**: 两个 worker 进程在线
  - **测试步骤**:
    1. 注册 worker-A 和 worker-B
    2. 创建多个待执行 Batch
    3. 观察 claim 行为
  - **预期结果**: Batch 按 slots 分配，单个 Batch 不被重复领取
  - **验证状态**: ⏳ 待验证
  - **备注**: 重点看并发一致性

- [ ] **TC-FR002-002**: 指定首选 worker 的任务只被目标 worker 领取
  - **前置条件**: 至少两台 worker 在线
  - **测试步骤**:
    1. 创建绑定 worker-A 的 Batch
    2. 观察 worker claim 结果
  - **预期结果**: 仅 worker-A 领取该 Batch
  - **验证状态**: ⏳ 待验证
  - **备注**: 验证 worker 绑定策略

### 异常场景（可选）

- [ ] **TC-FR002-ERR-001**: worker 心跳超时
  - **前置条件**: worker 已在线
  - **测试步骤**:
    1. 停止 worker 心跳
    2. 观察 controller 状态变化
  - **预期结果**: worker 被标记为不可用，但历史 Batch 信息保留
  - **验证状态**: ⏳ 待验证 / ⏭️ 跳过
  - **备注**: 一期可不自动迁移在途任务

## 需求 FR-003: Harbor 执行器接入

**需求描述**: worker 必须能调用 Harbor CLI 执行任务并归档原始结果。

### 正向场景（必选）

- [ ] **TC-FR003-001**: worker 构造 Harbor `docker` 命令并成功执行
  - **前置条件**: worker 安装 Harbor CLI 和 Docker
  - **测试步骤**:
    1. 发起 Harbor Batch
    2. 查看 worker 构造的执行命令
    3. 观察 Harbor 运行结束
  - **预期结果**: Harbor job 完成，原始 `jobs/` 目录归档到批次目录
  - **验证状态**: ⏳ 待验证
  - **备注**: 验证命令构造和路径布局

### 异常场景（可选）

- [ ] **TC-FR003-ERR-001**: Harbor CLI 不可用
  - **前置条件**: worker Harbor CLI 缺失或版本错误
  - **测试步骤**:
    1. 发起 Harbor Batch
    2. 观察 worker 执行结果
  - **预期结果**: Batch 失败，错误文本清晰可见
  - **验证状态**: ⏳ 待验证 / ⏭️ 跳过
  - **备注**: 验证诊断信息

## 需求 FR-004/005/006: 结果归一化、批次归档、失败诊断

**需求描述**: Harbor 结果必须被统一回收、归一化并形成不可变快照。

### 正向场景（必选）

- [ ] **TC-FR004-001**: Harbor 成功完成后生成 Batch 汇总和 CaseRun
  - **前置条件**: Harbor job 成功
  - **测试步骤**:
    1. 运行 Harbor Batch
    2. 解析 Harbor result
    3. 查看平台汇总结果
  - **预期结果**: Batch 和 CaseRun 指标正确，且可查看 Harbor 原始结果索引
  - **验证状态**: ⏳ 待验证
  - **备注**: 验证归一化层

- [ ] **TC-FR005-001**: 对同一 Run 重跑生成新 Batch
  - **前置条件**: 已存在一个完成的 Batch
  - **测试步骤**:
    1. 对同一 Run 再次发起执行
    2. 检查批次目录和状态
  - **预期结果**: 新旧 Batch 并存，旧快照不被覆盖
  - **验证状态**: ⏳ 待验证
  - **备注**: 验证归档不可变性

### 异常场景（可选）

- [ ] **TC-FR006-ERR-001**: Harbor 结果文件缺失
  - **前置条件**: 人为删除部分 Harbor 结果文件
  - **测试步骤**:
    1. 触发结果回收
    2. 查看 Batch 状态
  - **预期结果**: 平台记录缺失状态和错误文本，不伪造通过结果
  - **验证状态**: ⏳ 待验证 / ⏭️ 跳过
  - **备注**: 验证缺损场景处理
