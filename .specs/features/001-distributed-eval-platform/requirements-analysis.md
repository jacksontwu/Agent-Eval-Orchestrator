# 需求分析说明书: 分布式评测平台一期设计

**功能分支**: `001-distributed-eval-platform`  
**创建日期**: 2026-05-17  
**状态**: 草稿  
**输入**: 用户描述: "设计一个与 Harbor 解耦、支持固定 worker 与未来云弹性 worker 的分布式评测平台"

---

## 1. 简介

### 1.1 目的

本文档用于定义一个新的分布式评测平台一期需求。平台目标是支持固定几台机器组成 worker 集群，由 controller 统一编排 Harbor 评测任务，完成任务下发、执行状态观测、结果回收和批次归档，同时在架构上为后续云端弹性 worker 保留扩展能力。

### 1.2 范围

包含：

- 新平台的核心任务模型、调度模型、执行器抽象和结果归一化模型
- 固定 worker 集群的一期方案
- Harbor `docker` 执行器接入
- 运行结果回收、汇总、日志归档和批次视图
- 后续华为云/腾讯云动态 worker 的演进约束

不包含：

- 直接实现 Harbor 自定义云 Environment
- 一期内支持所有 Harbor benchmark adapter
- 复杂多租户 RBAC、计费和审计系统

### 1.3 假设和约束

| 类型 | 描述 |
|------|------|
| 假设 | Harbor 已可在 worker 本机通过 `docker` 模式稳定执行目标 benchmark |
| 假设 | 初期 worker 机器数量有限，通常为 2-10 台固定主机 |
| 假设 | 目标 benchmark 以 SWE-bench 类软件工程评测为主，但平台核心不绑定 `stage1/stage2` |
| 约束 | 平台核心必须与 Harbor 解耦，Harbor 只能作为可替换执行器 |
| 约束 | 一期优先可私有化部署，不能强制依赖公网 SaaS |
| 约束 | 所有批次结果必须可追溯、可回放、不可覆盖 |

### 1.4 术语与缩写

| 术语/缩写 | 全称 | 说明 |
|-----------|------|------|
| Controller | Control Plane | 负责 API、调度、状态持久化和资源管理的总控节点 |
| Worker | Execution Worker | 负责领取任务并执行评测的后台节点 |
| Executor | Execution Backend | 具体执行评测任务的后端实现，如 HarborExecutor |
| Run | 运行实体 | 用户可反复触发的一个任务运行对象 |
| Batch | 执行批次 | Run 的一次实际执行 |
| CaseRun | 用例执行记录 | 批次内某个实例/题目的一次执行记录 |

## 2. 系统上下文

### 2.1 系统定位

该功能是一个独立新平台的基础设计，用于承载后续 Harbor 评测、内部执行器、CodeBench 后处理以及弹性云 worker 等能力。它承担的是“平台层”的职责，而不是单一 benchmark 的执行脚本。

### 2.2 系统边界

平台内部：

- 任务模板、运行、批次、单题记录
- 用户、模型、执行器配置
- worker 注册、心跳、任务领取
- 运行结果归档与汇总

平台外部：

- Harbor CLI 和 Harbor task/dataset
- 本机 Docker 或未来云 sandbox
- 华为云/腾讯云 API
- 外部报告系统，如 CodeBench

### 2.3 系统架构图

```text
User/UI
  -> Controller API
      -> Scheduler / State Store / Artifact Index
      -> Worker Registry
      -> Executor Config Resolver
  -> Fixed Workers
      -> HarborExecutor (docker)
      -> Local runtime cache / docker images
  -> Shared Artifact Storage
      -> batch snapshots / logs / normalized results
```

## 3. 需求分析概述

### 3.1 需求来源

- 现有 `controller-worker` 平台与 Harbor 生态之间存在模型不一致
- 团队希望长期保留 Harbor 能力，同时避免平台被 Harbor 锁死
- 一期更关注“固定机器池稳定跑 Harbor 任务”，后续再扩展云弹性

### 3.2 需求目标

1. 用统一平台模型承载 Harbor 和未来其他执行器。
2. 一期支持固定 worker 集群稳定运行 Harbor `docker` 任务。
3. 保证每次执行都能回收结果、归一化指标、保存批次快照。
4. 为未来动态 worker 和云 API 接入保留清晰边界。

### 3.3 关键利益相关者

| 角色 | 职责 | 关注点 |
|------|------|--------|
| 平台管理员 | 部署 controller、worker、维护资源 | 系统稳定性、扩容方式、可运维性 |
| 评测使用者 | 创建任务、选择数据集与执行器、查看结果 | 易用性、结果清晰、可重跑 |
| 平台开发者 | 扩展执行器、支持新 benchmark | 抽象边界清晰、可插拔、低耦合 |
| 运维/基础设施团队 | 管理机器池、镜像、网络与存储 | 安全边界、故障恢复、成本与资源占用 |

## 4. 需求场景分析

### 4.1 业务场景

#### 4.1.1 业务流程图

```text
创建任务模板
  -> 创建 Run
  -> 选择实例集合并发起 Batch
  -> Controller 入队
  -> 合适 Worker 领取
  -> Executor 执行 Harbor Job
  -> 回收 Harbor 原始结果
  -> 归一化为 Batch / CaseRun 结果
  -> 展示汇总与题目级详情
```

#### 4.1.2 业务规则

| 规则编号 | 规则名称 | 规则描述 | 优先级 |
|----------|----------|----------|--------|
| BR-001 | 执行器解耦 | 平台核心状态不允许直接绑定 Harbor 专有字段 | 高 |
| BR-002 | Worker 绑定 | Run 首次被 worker 领取后，可配置为固定绑定策略 | 高 |
| BR-003 | 结果不可覆盖 | 新执行只能产生新 Batch，旧批次快照不可覆盖 | 高 |
| BR-004 | 归一化展示 | Harbor 原始结果必须转换成平台统一汇总结构 | 高 |
| BR-005 | 渐进扩展 | 固定 worker 是一期必选，动态云 worker 是后续扩展 | 中 |

### 4.2 用户操作场景与关键路径

#### 4.2.1 用户角色定义

| 角色ID | 角色名称 | 角色描述 | 权限范围 |
|--------|----------|----------|----------|
| UR-001 | 管理员 | 管理 worker、执行器、数据集、系统配置 | 全局管理 |
| UR-002 | 普通评测用户 | 创建任务模板、触发运行、查看结果 | 自己的任务空间 |

#### 4.2.2 用户操作场景

**场景 US-001: 在固定 worker 集群上发起 Harbor 评测** (优先级: P1)

| 项目 | 描述 |
|------|------|
| 场景描述 | 用户选择一个 Harbor 数据集、一个 agent/model 组合和一个执行器配置，发起一次 Harbor 评测批次 |
| 前置条件 | Controller、Worker 已运行，目标 worker 具备 Harbor 运行环境和数据集访问能力 |
| 操作步骤 | 1. 创建或选择任务模板<br>2. 创建 Run<br>3. 选择实例集合和目标执行器<br>4. 发起 Batch |
| 预期结果 | Batch 进入队列并最终在某个 worker 上完成执行，用户可看到汇总结果与题目级明细 |
| 异常处理 | 若 Harbor 启动失败、worker 异常或结果回收失败，平台保留错误文本和部分产物索引 |

**场景 US-002: 查看批次归档并定位失败题目** (优先级: P1)

| 项目 | 描述 |
|------|------|
| 场景描述 | 用户打开某次 Batch，查看总通过率、失败题目、日志入口和原始 Harbor 产物 |
| 前置条件 | Batch 已完成或失败 |
| 操作步骤 | 1. 打开 Run 详情<br>2. 进入某个 Batch<br>3. 查看 CaseRun 列表和日志索引 |
| 预期结果 | 用户能够区分平台归一化结果与 Harbor 原始结果，并追踪失败原因 |
| 异常处理 | 若某项原始产物不存在，平台显示缺失状态而不是猜测结果 |

**场景 US-003: 为未来弹性 worker 评估兼容性** (优先级: P2)

| 项目 | 描述 |
|------|------|
| 场景描述 | 平台开发者希望在不推翻核心模型的前提下，接入动态创建和回收的云 worker |
| 前置条件 | 一期固定 worker 方案已稳定 |
| 操作步骤 | 1. 复用 controller-worker 协议<br>2. 扩展 worker 生命周期管理<br>3. 接入云 API |
| 预期结果 | 无需改动核心任务模型即可扩展动态 worker |
| 异常处理 | 云扩缩容失败不影响现有固定 worker 方案 |

#### 4.2.3 关键路径分析

```text
创建 Batch -> 队列入库 -> Worker Claim -> Harbor 执行 -> 原始结果回收 -> 统一结果归档
```

| 路径编号 | 路径描述 | 关键节点 | 风险点 |
|----------|----------|----------|--------|
| KP-001 | Harbor 执行主路径 | claim、命令构造、结果归一化 | Harbor 结果格式变化 |
| KP-002 | 失败定位路径 | 错误上报、日志索引、CaseRun 状态 | 部分产物缺失 |
| KP-003 | 未来扩展路径 | 执行器抽象、worker 协议 | 核心模型被 Harbor 字段侵入 |

### 4.3 研发与维护阶段场景

#### 4.3.1 开发阶段场景

| 场景 | 描述 | 注意事项 |
|------|------|----------|
| 新增执行器 | 新增 Harbor 之外的执行器 | 只能依赖统一 Executor 协议 |
| 调整结果模型 | 增加新的归一化指标 | 不能破坏旧 Batch 快照可读性 |

#### 4.3.2 测试阶段场景

| 场景 | 描述 | 测试要点 |
|------|------|----------|
| Worker 领取任务 | 多 worker 并发领取 | 同一 Batch 不得重复领取 |
| 结果回收 | Harbor 执行成功/失败/中断 | 汇总和原始产物索引都要可观测 |

#### 4.3.3 运维阶段场景

| 场景 | 描述 | 运维要求 |
|------|------|----------|
| 固定机器运维 | Harbor、Docker、镜像缓存维护 | 支持节点下线和替换 |
| 共享存储维护 | 归档目录长期保存 | 需要容量治理与清理策略 |

### 4.4 兼容性分析

#### 4.4.1 版本兼容性

| 兼容类型 | 兼容要求 | 影响范围 |
|----------|----------|----------|
| 平台向前兼容 | 旧 Batch 快照在新版本平台中可读取 | 历史结果查看 |
| 执行器兼容 | Harbor 升级时平台可通过适配层局部调整 | 执行器插件 |

#### 4.4.2 平台兼容性

| 平台 | 兼容性要求 | 备注 |
|------|------------|------|
| Linux worker | 一期必须支持 | Harbor `docker` 执行主环境 |
| 云主机 | 后续扩展兼容 | 华为云/腾讯云 ECS 可作为 worker |

#### 4.4.3 数据兼容性

| 数据类型 | 兼容性要求 | 迁移策略 |
|----------|------------|----------|
| Batch 快照 | 新版平台可读取旧版快照 | 版本化 schema + 迁移器 |
| Executor 元数据 | Harbor 字段变化不影响平台核心表结构 | 存入扩展元数据区 |

## 5. 功能性需求分析

### 5.1 功能需求列表

| 需求编号 | 需求名称 | 需求描述 | 优先级 | 来源 |
|----------|----------|----------|--------|------|
| FR-001 | 统一任务模型 | 平台必须提供与执行器解耦的 TaskTemplate/Run/Batch/CaseRun 模型 | P1 | 长期演进需求 |
| FR-002 | 固定 worker 调度 | 平台必须支持固定 worker 集群注册、心跳、slots 和任务领取 | P1 | 一期运行需求 |
| FR-003 | Harbor 执行器接入 | 平台必须支持 Harbor `docker` 执行器构建命令、启动任务和收集原始结果 | P1 | 一期运行需求 |
| FR-004 | 结果归一化 | 平台必须把 Harbor 原始 job/trial 结果映射为平台统一汇总和 CaseRun 结果 | P1 | 结果展示需求 |
| FR-005 | 批次归档 | 每次执行必须产出不可变批次快照和产物索引 | P1 | 可追溯需求 |
| FR-006 | 失败可诊断 | 执行失败时必须保留错误文本、执行主机、日志索引和部分结果 | P1 | 运维需求 |
| FR-007 | 动态 worker 预留扩展 | 核心模型和协议必须支持未来动态创建 worker | P2 | 长期演进需求 |

### 5.2 功能需求详细说明

#### FR-001: 统一任务模型

| 项目 | 描述 |
|------|------|
| 需求描述 | 平台必须用统一模型描述任务模板、运行实体、执行批次和单题记录，不让 Harbor 语义直接污染核心数据库 |
| 验收标准 | 1. 核心表结构中不存在 Harbor 专有字段作为主状态字段<br>2. 执行器元数据位于扩展字段<br>3. UI 可以在不知道 Harbor 细节的情况下展示统一批次结果 |
| 关联场景 | US-001, US-002, US-003 |
| 依赖需求 | 无 |

#### FR-002: 固定 worker 调度

| 项目 | 描述 |
|------|------|
| 需求描述 | worker 必须支持注册、心跳、slots、领取任务、后台运行、结果上报 |
| 验收标准 | 1. controller 能识别在线/离线 worker<br>2. 每个 Batch 同时只会被一个 worker 领取<br>3. 可为 Run 指定首选 worker 或固定绑定 worker |
| 关联场景 | US-001 |
| 依赖需求 | FR-001 |

#### FR-003: Harbor 执行器接入

| 项目 | 描述 |
|------|------|
| 需求描述 | worker 必须能调用 Harbor CLI 运行任务，支持数据集路径、agent/model、并发、jobs 目录、运行命名等参数 |
| 验收标准 | 1. worker 可生成 Harbor 命令并执行<br>2. 命令失败会回传错误文本<br>3. Harbor 原始 `jobs/` 产物会被归档到批次目录 |
| 关联场景 | US-001 |
| 依赖需求 | FR-002 |

#### FR-004: 结果归一化

| 项目 | 描述 |
|------|------|
| 需求描述 | Harbor 运行结束后，平台必须提取 job 级和 trial 级结果，映射为 Batch 汇总和 CaseRun 记录 |
| 验收标准 | 1. Batch 有总量、完成量、成功量、失败量等指标<br>2. 每个 CaseRun 有明确状态、评分和产物索引<br>3. 原始 Harbor 结果和平台归一化结果可同时查看 |
| 关联场景 | US-002 |
| 依赖需求 | FR-003 |

#### FR-005: 批次归档

| 项目 | 描述 |
|------|------|
| 需求描述 | 每次执行必须生成只读快照，保存运行配置、统一结果、原始产物索引和时间信息 |
| 验收标准 | 1. 重跑不会覆盖旧批次<br>2. 可按日期和 Run 浏览历史批次<br>3. 快照支持未来 schema 迁移 |
| 关联场景 | US-002 |
| 依赖需求 | FR-004 |

#### FR-006: 失败可诊断

| 项目 | 描述 |
|------|------|
| 需求描述 | worker、Harbor、结果回收任一环节失败时，平台必须保留足够诊断信息 |
| 验收标准 | 1. Batch 和 CaseRun 都有错误文本字段<br>2. 至少保留 worker 日志和 Harbor 原始日志索引<br>3. UI 可区分执行失败与结果缺失 |
| 关联场景 | US-002 |
| 依赖需求 | FR-003, FR-004 |

#### FR-007: 动态 worker 预留扩展

| 项目 | 描述 |
|------|------|
| 需求描述 | 核心协议必须允许未来通过云 API 动态创建 worker 节点并加入集群 |
| 验收标准 | 1. worker 身份和生命周期与执行器分离<br>2. 动态 worker 不要求修改核心任务模型<br>3. 可在设计中明确接入点 |
| 关联场景 | US-003 |
| 依赖需求 | FR-001, FR-002 |


| 需求ID | 依赖需求 | 依赖类型 | 说明 |
|--------|----------|----------|------|

| FR-003 | FR-002 | 强依赖 | 执行器由 worker 调度触发 |

## 6. 非功能性需求分析

### 6.1 系统定位

目标用户为内部评测平台使用者和平台开发/运维团队。系统必须优先保证“长期可维护”和“结果可追溯”，而不是单次脚本式运行速度。

### 6.2 系统可靠性

| 指标 | 要求 | 说明 |
|------|------|------|
| 可用性 | Controller 支持后台持续运行 | 浏览器断开不影响任务 |
| 状态恢复 | controller/worker 重启后可恢复观测状态 | 依赖持久化存储 |
| 批次完整性 | 已结束批次结果不可被覆盖 | 归档为不可变快照 |
| 故障恢复策略 | worker 失联后任务保留状态并允许人工处理 | 一期先不做自动抢占恢复 |

### 6.3 安全隐私

| 方面 | 要求 |
|------|------|
| 身份隔离 | 用户任务空间和执行产物按用户隔离 |
| 密钥管理 | Harbor/模型/云 API 凭据不得写入批次公开快照 |
| 节点访问 | worker 只能通过 controller 协议写入平台状态 |

### 6.4 性能

| 指标 | 要求 |
|------|------|
| 调度延迟 | 空闲 worker 在一个轮询周期内可领取新任务 |
| 结果展示 | 批次汇总加载不依赖遍历所有原始大文件 |
| 可扩展性 | 支持 2-10 台固定 worker 的常规规模 |

### 6.5 可测试性

| 要求 | 说明 |
|------|------|
| 单元测试 | 状态机、调度逻辑、结果归一化、执行器命令构造 |
| 集成测试 | 单 worker Harbor mock、结果回收、批次归档 |
| 演练能力 | 可通过伪造 Harbor result 进行离线回放测试 |

### 6.6 合规与文档

| 要求 | 说明 |
|------|------|
| 中文文档 | 设计、操作和维护文档使用中文 |
| 变更记录 | 关键架构变更必须更新宪法和规格文档 |

## 7. 系统影响分析

### 7.1 影响列表

| 影响对象 | 影响程度 | 说明 |
|----------|----------|------|
| controller | 高 | 需要以新模型实现 API 和调度 |
| worker | 高 | 需要支持 HarborExecutor 和结果回收 |
| 存储布局 | 高 | 需要统一 Batch/CaseRun 产物布局 |
| 前端 | 中 | 需要摆脱 `stage1/stage2` 专有展示 |
| Harbor 依赖 | 中 | 需要约束 Harbor 仅作为执行器 |

### 7.2 影响分析

- 对现有 SWE-bench 专用平台的最大变化，是把 `stage1/stage2` 降为某种执行器私有语义，不再作为平台核心模型。
- 对 Harbor 的接入方式，是从“直接使用 Harbor 结果目录”变为“平台统一回收、归一化、归档”。
- 对未来云扩容的影响，是需要单独设计 worker 生命周期管理，但不需要重做任务模型。

### 7.3 风险评估

| 风险项 | 风险描述 | 缓解措施 |
|--------|----------|----------|
| R-001 | Harbor 结果格式升级导致解析失败 | 引入版本化 normalizer 和样本回放测试 |
| R-002 | 固定 worker 上 Harbor 环境不一致 | worker 启动时上报能力与版本，增加预检 |
| R-003 | 归一化结果丢失原始细节 | 保存 Harbor 原始 `jobs/` 目录索引和原始结果文件 |
| R-004 | 平台又被新执行器字段侵入 | 宪法强制执行 executorMetadata 边界 |
